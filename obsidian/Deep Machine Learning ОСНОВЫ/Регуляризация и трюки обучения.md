# Регуляризация, нормализация и трюки обучения

## Зачем это всё нужно

Нейронную сеть просто создать. Сложно — заставить её хорошо работать. Базовые техники, описанные здесь, отделяют "игрушечные примеры" от реальных моделей, которые работают на новых данных.

Три главные проблемы при обучении:
1. **Переобучение (overfitting)** — модель запомнила данные, а не выучила закономерности
2. **Затухание/взрыв градиентов** — градиенты становятся слишком маленькими (обучение встаёт) или слишком большими (loss улетает в NaN)
3. **Медленная сходимость** — сеть учится, но слишком долго

Каждая техника решает одну или несколько из этих проблем.

---

## 1. Dropout — отключаем нейроны случайно

### Проблема

Нейронная сеть с большим количеством параметров легко переобучается. Она начинает полагаться на конкретные нейроны или комбинации нейронов, которые "запомнили" обучающие данные. Если эти нейроны немного изменятся — сеть ошибается.

### Идея

Во время обучения случайно "выключаем" часть нейронов в каждом слое — обнуляем их выходы с вероятностью p. На каждом шаге (каждом батче) выключается разный набор нейронов. Это заставляет сеть не полагаться на отдельные нейроны, а учиться дублировать информацию — каждый нейрон учится быть полезным независимо от других.

На этапе инференса (проверки) Dropout отключается — работают все нейроны, но их выходы умножаются на (1-p), чтобы среднее значение совпадало со временем тренировки.

```python
import torch
import torch.nn as nn

class ModelWithDropout(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(256, 128)
        self.dropout = nn.Dropout(p=0.5)  # 50% нейронов выключается
        self.fc2 = nn.Linear(128, 10)

    def forward(self, x):
        x = F.relu(self.fc1(x))
        x = self.dropout(x)  # Dropout между полносвязными слоями
        x = self.fc2(x)
        return x

model = ModelWithDropout()

# Режим тренировки
model.train()
out = model(torch.randn(4, 256))  # Dropout работает

# Режим оценки
model.eval()
out = model(torch.randn(4, 256))  # Dropout отключён, все нейроны активны
```

### Dropout1d vs Dropout2d

- **nn.Dropout(p)** — обычный, для полносвязных слоёв. Выключает отдельные нейроны случайно.
- **nn.Dropout2d(p)** — для свёрточных слоёв. Выключает целые каналы (feature maps), а не отдельные пиксели. Почему? Потому что соседние пиксели в карте признаков сильно коррелируют — выключение одного пикселя мало что даёт. Выключение целого канала заставляет сеть не полагаться на одну карту признаков.

```python
# Dropout для свёрток — выключает целые каналы
dropout2d = nn.Dropout2d(p=0.3)
x = torch.randn(4, 64, 16, 16)  # [batch, channels, H, W]
out = dropout2d(x)  # ~30% каналов станут нулевыми для всего батча
```

### Когда использовать

- Dropout ставят **перед последним полносвязным слоем** (p=0.5)
- В свёрточных сетях dropout применяют реже, чем в полносвязных (там BatchNorm + weight decay работают лучше)
- Стандартные значения: p=0.5 для полносвязных, p=0.2-0.3 для свёрточных
- Если сеть маленькая и не переобучается — dropout не нужен

### Dropout — это ансамбль

Теоретически Dropout — это обучение огромного ансамбля сетей (каждая конфигурация выключенных нейронов — отдельная подсеть). На инференсе все эти подсети усредняются. Поэтому dropout так эффективен — это как иметь 1000 сетей, обученных на одних данных, но за цену одной.

## 2. Batch Normalization — стабилизируем распределения

### Проблема

Во время обучения веса каждого слоя меняются. Меняется и распределение выходов этого слоя. Следующий слой каждый раз видит данные с разным распределением — ему приходится постоянно подстраиваться. Это называется **Internal Covariate Shift** (смещение ковариат внутри сети).

На практике это выглядит так: обучение идёт рывками, loss скачет, приходится ставить маленький learning rate и долго ждать.

### Идея

Batch Normalisation нормализует выходы слоя внутри каждого батча: из каждого нейрона вычитаем среднее по батчу и делим на стандартное отклонение батча. Потом добавляем два обучаемых параметра — gamma (масштаб) и beta (сдвиг), чтобы сеть могла отменить нормализацию, если она мешает.

**Математика (одного нейрона):**

```
x_norm = (x - mean_batch) / sqrt(var_batch + eps)
y = gamma * x_norm + beta
```

eps = 1e-5 — чтобы не делить на ноль (если var = 0).

```python
# BatchNorm для свёрточных слоёв (нормализация по каналам)
bn2d = nn.BatchNorm2d(num_features=64)  # 64 канала

# BatchNorm для полносвязных слоёв
bn1d = nn.BatchNorm1d(num_features=128)  # 128 нейронов

# Как встраивается
class Net(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(256, 128)
        self.bn1 = nn.BatchNorm1d(128)     # BN после Linear
        self.fc2 = nn.Linear(128, 64)
        self.bn2 = nn.BatchNorm1d(64)      # BN после второго Linear
        self.fc3 = nn.Linear(64, 10)

    def forward(self, x):
        x = F.relu(self.bn1(self.fc1(x)))  # Linear → BN → ReLU
        x = F.relu(self.bn2(self.fc2(x)))
        x = self.fc3(x)
        return x
```

### Что нормализуется

**BatchNorm1d:** для каждого из 128 нейронов считаем среднее и std по батчу (по всем 32 примерам). Получается 128 средних и 128 std.

**BatchNorm2d:** для каждого из 64 каналов считаем среднее и std по батчу и по всем пространственным позициям (H×W). Получается 64 средних и 64 std.

```python
# Наглядно про нормализацию
x = torch.randn(32, 64, 16, 16)  # [batch, channels, H, W]
bn = nn.BatchNorm2d(64)

out = bn(x)
# out.mean() ≈ 0, out.std() ≈ 1 — для каждого канала отдельно
```

### Статистики на инференсе

На этапе тренировки BatchNorm использует среднее и std конкретного батча. На инференсе батч может быть размером 1 (один пример) — среднее будет равно самому примеру, а std — нулю. Это бессмысленно.

Поэтому BatchNorm **накапливает** бегущие средние (running mean, running var) во время тренировки. На инференсе использует их, а не статистики текущего батча:

```python
model.train()
# BN считает статистики по батчу, обновляет running_mean/var

model.eval()
# BN использует running_mean/var, накопленные за всё время обучения
# Статистики текущего батча не используются
```

### Что BatchNorm даёт на практике

1. **Высокий learning rate.** С BN можно ставить lr = 0.01-0.1 вместо 0.001. Обучение ускоряется в 5-10 раз.
2. **Меньше зависимость от инициализации.** Без BN если веса инициализировать неправильно — градиенты затухают. С BN сеть сама подстроит распределения.
3. **Лёгкая регуляризация.** Статистики батча — это шумные оценки (особенно на маленьких батчах). Этот шум работает как регуляризация, похожая на dropout. Часто с BN можно убрать или уменьшить dropout.
4. **Меньше проблем с затухающими градиентами.** Градиенты проходят через BN без затухания.

### Ограничения

- **Маленький батч (< 16) — BN работает плохо.** Статистики слишком шумные. Альтернатива: LayerNorm, GroupNorm, InstanceNorm.
- **RNN — BN работает плохо.** Для последовательностей лучше LayerNorm.
- **На инференсе поведение отличается от тренировки.** Нужно не забывать переключать `model.eval()`.

```python
# Проблема маленького батча
bn = nn.BatchNorm2d(64)
x = torch.randn(2, 64, 16, 16)  # batch=2 — слишком мало
# BN будет считать статистики по 2 примерам — очень шумно, обучение нестабильно

# Решение: GroupNorm — нормализация по группам каналов
gn = nn.GroupNorm(num_groups=8, num_channels=64)  # 8 групп по 8 каналов
x = torch.randn(2, 64, 16, 16)
out = gn(x)  # работает отлично при любом batch_size
```

### Порядок слоёв: Conv → BN → ReLU

Стандартный паттерн:

```
Conv2d → BatchNorm2d → ReLU
Linear → BatchNorm1d → ReLU
```

Почему BN до ReLU? Потому что нормализация работает лучше до нелинейности: распределение до ReLU более симметрично (ближе к нормальному), после ReLU — половина значений 0. Но на практике разница небольшая, встречается и `Conv → ReLU → BN`.

## 3. Weight Decay (L2-регуляризация) — штраф за большие веса

### Проблема

Большие веса — признак того, что сеть "выкручивается", чтобы запомнить конкретные примеры. Маленький шум в данных при больших весах приводит к большому изменению выхода.

### Идея

Добавляем к функции потерь штраф за большие веса: `loss = original_loss + lambda * sum(w²)`. lambda — коэффициент регуляризации.

```python
# В оптимизаторе — параметр weight_decay
optimizer = torch.optim.SGD(model.parameters(), lr=0.01, weight_decay=1e-4)
# weight_decay = lambda — чем больше, тем сильнее штраф

# Это эквивалентно тому, что на каждом шаге веса чуть-чуть уменьшаются:
# w = w - lr * (grad + lambda * w)
```

### Когда и сколько

- Стандартное значение: `weight_decay = 1e-4`
- Для маленьких сетей (нет переобучения) — 0
- Для больших сетей — 1e-5 до 1e-3
- С BN weight_decay можно ставить меньше — BN уже даёт регуляризацию

### L1 vs L2 регуляризация

- **L2 (weight_decay):** штраф = sum(w²). Веса становятся маленькими, но не нулевыми. Стандартный выбор.
- **L1:** штраф = sum(|w|). Веса становятся нулевыми — разреживание. Используется, когда нужно отобрать признаки.

```python
# L2 — через weight_decay в оптимизаторе (самый простой способ)
optimizer = torch.optim.Adam(model.parameters(), lr=0.001, weight_decay=1e-4)

# L1 — нужно добавить вручную в функцию потерь
def l1_penalty(model):
    return sum(p.abs().sum() for p in model.parameters())

loss = criterion(pred, y) + 1e-5 * l1_penalty(model)
```

## 4. Learning Rate и его планирование

### Проблема

Learning rate — самый важный гиперпараметр:
- Слишком большой: loss расходится (уходит в NaN)
- Слишком маленький: обучение идёт годами
- Правильный: loss плавно падает, но к концу обучения можно было бы ускориться

### Решение: планировщики (schedulers)

Идея: начинаем с относительно большого lr (быстрый прогресс), потом уменьшаем (тонкая настройка).

```python
import torch.optim as optim
from torch.optim.lr_scheduler import StepLR, ReduceLROnPlateau, CosineAnnealingLR

model = nn.Linear(10, 2)
optimizer = optim.SGD(model.parameters(), lr=0.1)

# Вариант 1: StepLR — уменьшаем lr каждые N эпох
scheduler = StepLR(optimizer, step_size=30, gamma=0.1)
# lr = 0.1 на эпохах 1-30, 0.01 на 31-60, 0.001 на 61-90

# Вариант 2: ReduceLROnPlateau — уменьшаем, когда loss перестал падать
scheduler = ReduceLROnPlateau(optimizer, mode='min', factor=0.5, patience=5)
# Если loss не уменьшался 5 эпох — умножаем lr на 0.5

# Вариант 3: CosineAnnealing — плавное уменьшение по косинусу
scheduler = CosineAnnealingLR(optimizer, T_max=50)
# lr плавно идёт от 0.1 до 0 за 50 эпох

# Применение
for epoch in range(100):
    train_one_epoch()
    val_loss = validate()

    # Вариант 1 и 3: просто шагаем
    scheduler.step()

    # Вариант 2: передаём метрику
    scheduler.step(val_loss)
```

### One Cycle Policy

Самый современный подход: сначала увеличиваем lr (разогрев), потом уменьшаем. Идея — "разогреть" оптимизатор, выйти из плохих локальных минимумов, потом плавно сойтись в хороший:

```python
from torch.optim.lr_scheduler import OneCycleLR

optimizer = optim.SGD(model.parameters(), lr=0.01)
scheduler = OneCycleLR(
    optimizer,
    max_lr=0.1,          # пиковый learning rate
    steps_per_epoch=len(train_loader),
    epochs=10
)

# lr:  0.01 → плавно → 0.1 (разогрев 30% эпохи) → плавно → ~0
```

### Найти хороший lr: lr-range test

Простой трюк: начинаем с очень маленького lr (1e-7) и увеличиваем его на каждом батче, пока loss не перестанет падать. Оптимальный lr — тот, где loss падает быстрее всего.

```python
# Упрощённая версия lr-finder
lr = 1e-7
optimizer = optim.SGD(model.parameters(), lr=lr)

for batch in range(100):
    loss = train_step(model, optimizer, batch)
    lr *= 1.5  # увеличиваем lr на каждом шаге
    for param_group in optimizer.param_groups:
        param_group['lr'] = lr

    print(f"lr={lr:.2e}, loss={loss:.4f}")
    if loss > prev_loss * 2:  # loss начал расти — стоп
        break
```

## 5. Инициализация весов

### Проблема

Если все веса инициализировать нулями — все нейроны будут обновляться одинаково (симметрия). Если случайными числами — слишком большие веса дадут большие выходы, насыщение активаций, затухание градиентов.

### Решения

```python
import torch.nn.init as init

# Xavier (Glorot) — для tanh, sigmoid (классика)
# w ~ Uniform(-sqrt(6/(n_in + n_out)), sqrt(6/(n_in + n_out)))
init.xavier_uniform_(layer.weight)

# Kaiming (He) — для ReLU (современный стандарт)
# w ~ Normal(0, sqrt(2/n_in))
init.kaiming_normal_(layer.weight, mode='fan_in', nonlinearity='relu')

# В PyTorch это часто встроено в слой:
conv = nn.Conv2d(3, 64, 3)
# По умолчанию — Kaiming Uniform для Conv2d и линейных слоёв
```

**Правило:** для ReLU используйте Kaiming, для tanh/sigmoid — Xavier. С BatchNorm инициализация менее критична, но правильная помогает с первых эпох.

## 6. Gradient Clipping — обрезаем градиенты

### Проблема

Иногда градиент становится очень большим (из-за специфичного батча с выбросами). Вес делает огромный шаг — loss "взлетает" до небес, может уйти в NaN. Особенно часто в RNN и трансформерах.

### Решение

Обрезаем градиент по норме (или по значению), прежде чем оптимизатор сделает шаг:

```python
# По норме (clip_grad_norm_) — самый популярный вариант
# Если норма градиента > max_norm — масштабируем все градиенты
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# По значению (clip_grad_value_) — обрезаем каждый градиент отдельно
torch.nn.utils.clip_grad_value_(model.parameters(), clip_value=0.5)

# Применение в цикле обучения
for batch_x, batch_y in dataloader:
    pred = model(batch_x)
    loss = criterion(pred, batch_y)

    optimizer.zero_grad()
    loss.backward()

    # Gradient clipping — перед шагом оптимизатора
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)

    optimizer.step()
```

**max_norm=1.0** — стандартное значение. Если loss всё ещё взлетает — уменьшите до 0.5. Если обучение слишком медленное — увеличьте до 5.0.

## 7. Label Smoothing — смягчаем метки

### Проблема

Стандартные метки — one-hot: `[0, 0, 1, 0, 0]`. Сеть учится выдавать уверенность 100% для правильного класса. Это может приводить к переобучению и слишком самоуверенным предсказаниям.

### Решение

Вместо one-hot используем "размазанные" метки: `[0.025, 0.025, 0.9, 0.025, 0.025]`. То есть немного уверенности забираем у правильного класса и распределяем между неправильными.

```python
def label_smoothing(y_true, num_classes, smoothing=0.1):
    """y_true: [batch] с индексами классов"""
    confidence = 1.0 - smoothing
    smooth_labels = torch.full(
        (len(y_true), num_classes),
        smoothing / (num_classes - 1)
    )
    smooth_labels.scatter_(1, y_true.unsqueeze(1), confidence)
    return smooth_labels

# Использование
y = torch.tensor([2, 0, 3])  # индексы классов
smooth_y = label_smoothing(y, num_classes=10, smoothing=0.1)
# tensor([
#   [0.011, 0.011, 0.9, 0.011, ...],
#   [0.9, 0.011, 0.011, 0.011, ...],
#   ...
# ])
```

Или через CrossEntropyLoss с label_smoothing (PyTorch 1.10+):

```python
criterion = nn.CrossEntropyLoss(label_smoothing=0.1)
```

**Когда использовать:** почти всегда. 0.1 — стандартное значение. Особенно полезно при большом количестве классов.

## 8. Early Stopping — остановка вовремя

### Проблема

Если обучать слишком долго, модель переобучится: loss на тренировке будет падать всё ниже, а на валидации — начнёт расти.

### Решение

Следим за loss на валидации и останавливаемся, если он не улучшался заданное количество эпох (patience). Сохраняем лучшую модель по валидации:

```python
class EarlyStopping:
    def __init__(self, patience=10, min_delta=0.001):
        self.patience = patience
        self.min_delta = min_delta
        self.best_loss = float('inf')
        self.counter = 0
        self.best_model = None

    def check(self, model, val_loss):
        if val_loss < self.best_loss - self.min_delta:
            self.best_loss = val_loss
            self.counter = 0
            self.best_model = model.state_dict().copy()
            return False  # не останавливаемся
        else:
            self.counter += 1
            if self.counter >= self.patience:
                # Восстанавливаем лучшую модель
                model.load_state_dict(self.best_model)
                return True  # останавливаемся
            return False

# Использование
early_stop = EarlyStopping(patience=10)

for epoch in range(1000):
    train_loss = train_epoch(model, train_loader)
    val_loss = validate(model, val_loader)
    print(f"Epoch {epoch}: train={train_loss:.4f}, val={val_loss:.4f}")

    if early_stop.check(model, val_loss):
        print("Early stopping! Восстановлена лучшая модель.")
        break
```

## 9. Анализ градиентов — диагностика обучения

Иногда сеть не обучается, и непонятно почему. Посмотрите на градиенты:

```python
def check_gradients(model):
    total_norm = 0
    for p in model.parameters():
        if p.grad is not None:
            param_norm = p.grad.norm(2)
            total_norm += param_norm.item() ** 2
    total_norm = total_norm ** 0.5

    # Считаем долю "мёртвых" градиентов (Zero gradients)
    zero_grads = 0
    total_params = 0
    for p in model.parameters():
        if p.grad is not None:
            total_params += p.numel()
            zero_grads += (p.grad == 0).sum().item()

    print(f"Норма градиента: {total_norm:.4f}")
    print(f"Доля нулевых градиентов: {zero_grads / total_params:.2%}")
```

**Что искать:**
- Норма градиента ≈ 0 → затухание градиентов (смените активацию, добавьте BN)
- Норма градиента > 100 → взрыв градиентов (добавьте gradient clipping)
- 90%+ градиентов = 0 → мёртвые ReLU (слишком большой lr, попробуйте LeakyReLU)

## 10. Шпаргалка: что и когда применять

| Техника | Решает проблему | Когда применять | Стандартные параметры |
|---------|----------------|-----------------|----------------------|
| **Dropout** | Переобучение | Полносвязные слои больших сетей | p=0.5 (FC), p=0.2-0.3 (Conv) |
| **BatchNorm** | Медленная сходимость, стабильность | Почти всегда, кроме совсем маленьких сетей | — |
| **Weight Decay** | Переобучение | Всегда, если есть переобучение | 1e-4 (SGD), 0 (Adam — встроен) |
| **Gradient Clipping** | Взрыв градиентов | RNN, трансформеры, глубокие сети | max_norm=1.0 |
| **Label Smoothing** | Переобучение, самоуверенность | Классификация с большим числом классов | 0.1 |
| **Early Stopping** | Переобучение | Всегда | patience=10 |
| **LR Scheduler** | Медленная сходимость в конце | Почти всегда | ReduceLROnPlateau или Cosine |
| **Kaiming Init** | Медленная сходимость в начале | Для ReLU сетей | По умолчанию в PyTorch |

### Полный шаблон обучения

```python
model = MyModel()
optimizer = torch.optim.Adam(model.parameters(), lr=0.001, weight_decay=1e-4)
scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau(
    optimizer, mode='min', factor=0.5, patience=5
)
early_stop = EarlyStopping(patience=10)
criterion = nn.CrossEntropyLoss(label_smoothing=0.1)

for epoch in range(1000):
    # Тренировка
    model.train()
    for x, y in train_loader:
        pred = model(x)
        loss = criterion(pred, y)
        optimizer.zero_grad()
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()

    # Валидация
    model.eval()
    val_loss = validate(model, val_loader)

    # Планирование
    scheduler.step(val_loss)  # ReduceLROnPlateau требует метрику

    # Остановка
    if early_stop.check(model, val_loss):
        break
```
