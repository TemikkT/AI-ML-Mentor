## Создание таблиц, типы данных и связи между таблицами

Сначала разберёмся с тем, как создавать таблицы, что такое Первичные и внешние ключи в ней. Пример создания таблицы:

```SQL
CREATE TABLE band(
	band_id INTEGER,
	name VARCHAR,
	year SMALLING,
	comment VARCHAR
);
```

Создаём таблицу band с 4 колонками. Так же указаны названия колонок и их типы данных. Типов данных очень много, рассмотрим самые популярные!
Числовые типы:
SMALLING - 2 байта на число 
INTEGER - 4 байта на число
BIGINT - 8 байт на число
![[Pasted image 20260530130913.png]]

Строковые типы:
CHAR(n) или CHARACTER(n) - для текста фиксированной длины
VARCHAR(n) или CHARACTER VARYING(n) - для текста переменной длины
![[Pasted image 20260530130955.png]]

Тут важно заметить в SELECT, если мы хотим обозначить опостроф в строке, то делаем это при помощи двух верхних кавычек (все одинарные) 'It''s a pencil'. Если же нам надо объединить строки, то делаем это так: 'Hello' | | 'World'

Следующие типы данных Дата, Время, Дата + Время
DATE - Дата
TIME - Время
TIMESTAMP - Дата + Время
TIMESTAMP WITH TIME ZONE - С часовым поясом
![[Pasted image 20260530131226.png]]

Таблицы так же могут ссылаться друг на друга, как например на следующем рисунке
![[Pasted image 20260530132040.png]]

Тут таблица album ссылается на таблицу band по ключу band_id.
В колонке band колонка band_id должно принимать всегда уникальные значения, это айди группы, было бы странно если бы у двух групп был один айди, такое условие называется Условием Первичного ключа, то есть данная колонка является PRIMARY KEY если она уникальна и не содержит значения NULL. А в таблице Album должны быть в band_id только те значения, которые уже есть в таблице band, чтобы мы не ссылались на несуществующую группу. Это условие называется условием Внешнего ключа - FOREIGN KEY (Ссылается на первичный ключ в другой таблице). Первичный ключ вообще не обязательно указывать при создании таблицы, она будет создана и без него, но это дурной знак, лучше казать, пример:
```SQL
CREATE TABLE band(
	band_id INTEGER NO NULL
			PRIMARY KEY,
	name VARCHAR,
	year SMALLING,
	comment VARCHAR
);
```
Так таблица сама будет проверять уникальность и отсутствие NULL.
Так же и для того чтобы ссылаться, создание внешнего ключа:
```SQL
CREATE TABLE album(
	album_id INTEGER NO NULL
			PRIMARY KEY,
	name VARCHAR,
	band_id INTEGER
		REGERENCES band(band_id),
	year SMALLING
);
```

Теперь поговорим о том, какие связи бывают у таблиц. Первая связь это 

1:M - это Один ко Многим. Когда в первой таблице ключ встречается лишь 1 раз, но во второй таблице она может встречаться много раз

M:1 - Одно и тоже, просто таблицы поменяли местами.

1:1 - Одна строка одной таблицы соответствует одной строке другой таблицы

М:М - Многие строки первой таблицы соответствуют многим строкам второй таблицы

Зачастую так не делают, а добавляют третью связующую таблицу между ними, чтобы получить 1:M  -  M:1

## SELECT вся база написания запросов

Базовый SELECT выглядит так:
```SQL
SELECT band_id, name, year, comment
FROM band
```

В самом SELECT мы пишем те колонки, которые мы хотим вернуть при помощи запроса, в а FROM пишем название самой таблицы откуда мы берём эти колонки, так же если мы хотим вернуть вообще все колонки можно просто написать SELECT * это будет означать - Верни всё из таблицы. Так же 

```SQL
SELECT band_id, name, year, comment
FROM band as b
```
можно создавать алиасы как колонок, так и таблицы в целом. Тут написано FROM band as b - это означает, что далее в запросе я буду обращаться к этой таблице просто как b.
![[Pasted image 20260530134902.png]]

Так же, мы можем создавать свои колонки и выводить их из существующих, простой пример:
```SQL
SELECT band_id, 
	   name,
	   upper(name) as u_name 
FROM band
```
![[Pasted image 20260530135011.png]]

Мы вывели новую колонку u_name в которой названия с заглавной буквы, конечно есть очень много различных выражений, это просто пример для одной из них, для них будет отдельный раздел, как с ними работать правильно и какие бывают. 
Это была фильтрация колонок, теперь поговорим об фильтрации строк

Посмотрим на один базовый пример:
```SQL
SELECT band_id,name
FROM band
WHERE band_id = 93
```
![[Pasted image 20260530135204.png]]

При помощи такой функции как WHERE мы как бы прямо говорим, верни там то ГДЕ (WHERE) айди группы равно 93. Если бы было более одной строки, то все они попали бы в результат. 
Есть ещё один вариант фильтрации строк, это аналог head() из пандаса, а именно LIMIT:
```SQL
SELECT band_id,name
FROM band
LIMIT 10
```
Так мы выведем первые 10 строчек таблицы, допустим просто хотим посмотреть как они выглядят (это можно по крутому использовать, допустим отфильтровать какую то колонку и вывести LIMIT 1, так выведем максимум или минимум)

В WHERE можно ставить конечно же не только одно условие:
```SQL
SELECT band_id,name
FROM band
WHERE 
	   band_id = 93
	OR band_id = 192
```
Так мы получим уже два фильтра, по айди 93 и 192.
Если нужно чтобы оба условие выполнялись одновременно:

```SQL
SELECT *
FROM band
WHERE 
	    year < 1980
	AND comment <>''
```
Тут вы берём строки, в которые год ниже 1980 и указан НЕ пустой комментарий (<> означает НЕ РАВНО, крч это != в мире SQL). Все эти условия можно мешать друг с другом. Но тут важно учесть последовательность:
```SQL
SELECT *
FROM band
WHERE band_id = 303
	OR (band_id = 93 AND year = 1981)
```
Допустим такое. Сначала она найдёт то, где айди 303 отдельно, а потом уже будет искать второе условие, которое находится в скобках, где айди 93 И год равен 1981
![[Pasted image 20260530135921.png]]

Но если мы расставим скобки по другому:
```SQL
SELECT *
FROM band
WHERE (band_id = 303
	OR band_id = 93) AND year = 1981
```
Теперь он сначала найдёт строки где айди равен 303 или 93, потом уже среди них попытается найти где год равен 1981 и выведет их
![[Pasted image 20260530140035.png]]

Но если скобки не указаны, то в работу идёт стандарт SQL в котором говорится. Сначала выполняем AND, а потом уже OR, то есть выведет нам первый результат, обведёт сам скобками оператор AND. Но лучше всегда использовать скобки для читабельности.

А если нам надо вывести много различных значений айди, будет не особо красиво выглядеть допустим 3 подряд OR в запросе, на это тоже есть ответ, использовать IN (опять же напоминает .isin в пандасе):

```SQL
SELECT band_id, name
FROM band
WHERE 
	band_id IN (303, 93, 192)
```
![[Pasted image 20260530140354.png]]

То есть мы выводим все айди, которые есть в нашем кортеже, которые мы указали. То есть мы прямо говорим ВЫВЕДИ С АЙДИ КОТОРЫЕ СОДЕРЖАТЬСЯ В ЭТОМ КОРТЕЖЕ. Так же к этому можно добавить NOT и получится NOT IN, обратная вещь.

Так же очень важно подметить, что в IN можно вставлять не только базовые кортежи для сортировки, но так же и целые ПОДЗАПРОСЫ, в которых уже мы выводим то что нам надо увидеть. допустим у нас есть колонка:
![[Pasted image 20260530141107.png]]

и мы можем её вывести как подзапрос в IN чтобы получить нужный результат:
```SQL
SELECT band_id, name
FROM band
WHERE 
	band_id IN (SELECT id
				FROM list1
				)
```

Ещё один очень важный оператор для WHERE, это BETWEEN, а именно он указывает границы со скольки и до скольки нам нужно взять числа, пример с годом:
```SQL
SELECT band_id, name, year
FROM band
WHERE 
	year BETWEEN 1970 and 1981
```
Данный запрос аналогичен более простому, который будет понятен всем:
```SQL
SELECT band_id, name, year
FROM band
WHERE 
	  year >= 1970
  and year <= 1981
```

Так же ещё вот небольшие вещи с NOT, чтобы не писать <>, просто пишем NOT перед нашей фильтраций и получается обратный запрос
![[Pasted image 20260530141543.png]]

___
Так же думаю очень важным будет поговорить более подробно про вложенные запросы, где мы подаём целый запрос в WHERE. Такие вложенные запросы можно подавать много куда и с их помощью можно творить очень удивительные вещи, главное додуматься до этого. Вот несколько примеров:
```SQL
SELECT *
FROM 
	 (SELECT band_id, name
	 FROM band
	 WHERE year >=1980
	 ) as p
WHERE
	p.band_id BETWEEN 10 and 20
```

В данном запросы мы положили внутренний запрос в FROM, то есть наш основной запрос будет брать таблицу, которую мы вывели во вложенном запросе. Добавляем к нему алис "as p" чтобы к нему было удобно обращаться и в WHERE указываем так же условие которые нам нужны. 
Такие подзапросы очень полезны и удобны в сложных запросах, когда данные берутся из нескольких таблиц. К этому мы вернёмся когда будем рассматривать JOIN (объединение таблиц)

## Работа со значениями NULL

Это невероятно важная тема, потому что NULL будет встречаться повсюду, и сегодня научимся с ним работать, как его избегать, обрабатывать и вовремя замечать.
Это неизвестной, отсутствующее значение, потеря данных. Это НЕ НУЛЕВОЕ значение, это просто потеря данных. Для проверки колонки на NULL используется специальный синтаксис, а именно:
```SQL
SELECT id, name, parent_id
FROM music_instrument
WHERE parend_id IS NULL
```

Так нам выведет все строки, где parent_id является NULL. Но не стоит писать = NULL. Ведь NULL это неопределённость, с чем мы сравниваем, непонятно, поэтому такое ничего не выведет, всегда с NULL нужно IS.

Чтобы заменить NULL на какие то простые значение, можно использовать функцию COALESCE, в ней указываются 2 параметра, первый это колонка в которой будет замена, а второй, это то - на что мы поменяем наш NULL
```SQL
SELECT id,
	   name,
	   paren_id,
	   COALESCE(parent_id, -1) as parend_id2
FROM music_instrument
```
Так мы создадим новую колонку в таблице при выводе, в которой все NULL в parend_id заменяться на -1.
Так же очень важно учесть, что если мы складываем что либо с NULL. То мы тоже получим NULL. Сложение с неизвестным тоже является неизвестным, тут надо быть осторожным. Если хотим что то сложить и сильно боимся из-за NULL в данных, или мы знаем что оно там есть, но его надо как то опустить не заменяя, то удобно будет такой вариант:
![[Pasted image 20260530145025.png]]

Поговорим как IN и NOT IN работает с NULL, что тут может быть. Вернёмся к данному запросу:
```SQL
SELECT band_id, name
FROM band
WHERE 
	band_id IN (SELECT id
				FROM list1
				)
```
Но теперь в нашей таблице list1 появился NULL, как запрос поведёт себя теперь?
![[Pasted image 20260530145320.png]]

Запрос вернётся абсолютно те же значение, что и до этого. Ведь наш запрос выше, равен запросу:
```SQL
SELECT band_id, name
FROM band
WHERE 
	band_id = 303
 OR band_id = 93
 OR band_id = NULL
```
Нооо, как мы помним, нельзя ставить = с NULL, это просто не работает, без ошибок, без всего, это просто сравнение с неизвестным и оно не работает, поэтому вывод никак не изменится. Но лучше просто добавить в подзапрос фильтр, что мы убираем IS NOT NULL:
```SQL
SELECT band_id, name
FROM band
WHERE 
	band_id IN (SELECT id
				FROM list1
				WHERE id IS NOT NULL
				)
```

Вот так будет лучше для всех.

## Агрегация и группировка данных

Для начала с простого, допустим мы хотим вывести количество строк в таблице, просто число вывести. Такой запрос пишется подобным образом, используем функцию агрегации, а именно:
```SQL
SELECT COUNT(*)
FROM band
```
Функция агрегации тут это COUNT, а звёздочка это означает что выводим просто всё. Тут мы считаем количество строк во всей таблице, тк у нас нету никаких условий. Допустим хотим посчитать кол-во альбомов группы с айди 93:
```SQL
SELECT COUNT(*)
FROM album
WHERE band_id = 93
```

А если мы хотим вывести не только просто число, но и сам айди рядом, то есть количество чего мы вывели типо.

```SQL
SELECT 93 as band_id,
		COUNT(*)
FROM album
WHERE band_id = 93
```

Используем константу 93 и записываем её в новую созданную колонку band_id

А если у нас много таких номеров и мы хотим вывести вообще все, а не вот так по одному. Хотим единую таблицу со всеми группами и кол-во их альбомов

| band_id | COUNT |
| ------- | ----- |
| 93      | 2     |
| 192     | 3     |
Тут и пригодится такая вещь, как группировка строк, GROUP BY (опять же 1 в 1 Pandas с его df.groupby):

```SQL
SELECT band_id,
		COUNT(*)
FROM album
GROUP BY band_id
```

GROUP BY band_id - это и есть группировка по колонке band_id. А в SELECT мы как раз и указываем, что показываем тот самый сгрупированный band_id и количество каждой группы.

```SQL
SELECT year,
		COUNT(*)
FROM album
GROUP BY year
```

Ещё пример. 
**Важное уточнение, ТО что мы пишем в SELECT должно находится в GROUP BY, либо агрегационный функции, то есть нельзя записать и band_id и year в SELECT когда группировка идёт по только band_id. Нужны обе, группируй сразу две**
```SQL
SELECT year, band_id,
		COUNT(*)
FROM album
GROUP BY year, band_id
```

Так же важно поговорить о такой вещи как Having. Изначально её и вовсе можно перепутать с WHERE, но нельзя. В WHERE мы работаем только с настоящими колонками таблицы, то что там уже всегда было, нам запрещено ставить условия для агрегированных функций. Происходит это потому что SQL запрос сначала смотрит на WHERE и сортирует как надо, и потом уже делает группировку и агрегацию, поэтому на первом этапе он не видит никаких новых агрегированных столбцов. 
Поэтому для такого придумали такую вещь как HAVING, в котором как раз мы и делаем условие для агрегированных функций, вот пример:

```SQL
SELECT band_id, COUNT(*)

FROM album
WHERE band_id IN (93, 192)

GROUP BY band_id
HAVING COUNT(*) = 11
```

Вот это правильный запрос. Мы делаем фильтрацию WHERE для изначальной таблицы, чтобы выбрать только 2 значений айдишника, а потом уже идёт группировка именно по этим двум айдишникам и далее агрегаций для них, а потом уже полученную таблицу мы фильтруем при помощи HAVING.

Из этого урока выходит один ооочень важный вывод, а именно то, что - Порядок написания SQL запросов не совпадает с порядком их выполнения. Вот как это выглядит на самом деле.

1. FROM - выбор данных, выбираем, берём таблицу с которой мы будем работать (Тут же выполняются соединения JOIN)
2. WHERE - обозначаем фильтрацию для таблицы
3. GROUP BY - далее группировка по нужной колонке по отфильтрованной таблице
4. HAVING - после получения сгруппированной таблицы происходит фильтрация агрегированных значений при помощи HAVING
5. SELECT - выбираются нужные там колонки уже в считайте что готовой таблице, да да да, сначала строится таблица, а потом уже из неё берутся все колонки, это вроде очевидно, но это очень важно понимать при написании запросов. Так же тут вычисляются все агрегатные функции
6. ORDER BY - сортировка результата
7. LIMIT - ограничиваем число строк для показа

так же стоит показать и все другие функции агрегации, помимо COUNT. Что с ними делать и как круто их использовать можно говорить очень много, поэтому просто оставлю это тут
![[Pasted image 20260531222257.png]]

DISCTINCT в свою очередь выводит все уникальные значения колонки и если использовать её вместе с COUNT, то получим количество уникальных значений.

Так же я уже упоминал такую вещь, как Сортировка данных, а именно ORDER BY. 
```SQL
SELECT album_id, name, band_id, year
FROM album
ORDER BY year DESC, band_id
```
Допустим в этом примере происходит сортировка по 2 значениям. Сначала для YEAR с указанием DESC, это означает сортировку по убыванию, а потом уже band_id.

## Соединение таблиц

#### INNER JOIN

![[Pasted image 20260601005551.png]]


Представим у нас есть 2 таблицы, и мы хотим их склеить в одну единую таблицу по какому то общему признаку. INNER JOIN это именно та операция которая берёт только совпадающие строки из обеих таблиц. Если смотреть кружками, то выглядит она так
![[Pasted image 20260601005848.png]]

Допустим вот пример - У нас есть первая таблица Клиентов и вторая таблица Заказы.
![[Pasted image 20260601005919.png]]
![[Pasted image 20260601005925.png]]

Мы хотим получить таблицу, где будет видно какой человек что именно купил. Поэтому мы будем склеивать таблицы по их общему ключу, а именно CustomerID.
```SQL
SELECT
	Customer.name
	Orders.Product
FROM Customers
INNER JOIN 
	Orders ON Customers.CustomerID = Orders.CustomerID
```

Тут очень важно заметить как мы записываем SELECT. Не забываем как строится весь запрос sql. Сначала читается FROM, берётся таблица которую мы указали, Потом происходит объединение таблиц, и только потом из общей таблицы мы переходим в SELECT чтобы выбрать с чем работать.
Почему мы выбрали именно Customers в FROM? Это просто начальная точка, порядок тут не особо важен, можно было записать и:
```SQL
SELECT
	Customer.name
	Orders.Product
FROM Orders
INNER JOIN 
	Customer ON Orders.CustomerID = Customer.CustomerID
```

в INNER JOIN же мы пишем то, какую таблицу мы присоединяем к нашей, то есть если мы выбрали Orders как начальную, то склеиваем другую.
Так же очень важно! Далее мы указываем условие стыковки, а именно **Когда айди из одной колонки равно айди другой колонке**

На выходе мы получаем вот это:
![[Pasted image 20260601010537.png]]

Алисы у нас две, тк в таблице Orders у неё было два заказа. Виктор в свою очередь и вовсе не попал в таблицу, ведь у него нет заказов.

#### LEFT OUTER JOIN и RIGHT OUTER JOIN
![[Pasted image 20260601015732.png]]

Опять же у нас есть две таблицы. Правая и Лева. Суть такова: мы берём все строки из левой таблицы и прикрепляем к ним данные из правой таблицы там, где они есть. Если для какой то таблицы слева нету пары в правой таблице, то заполняй все колонки из правой таблицы значениями NULL.

Левая таблица Главная, она сохраняет все свои записи, даже они не соответствуют условию ON (как мы помним, в прошлом INNER JOIN, ему важно было чтобы строки совпадали и были в обоих. Тут же нам важно чтобы они были только в левой, из правой мы берём что есть и прикрепляем, если пары нету возвращаем NULL).

По сути LEFT JOIN отвечает на вопросы: Показать всё из таблицы А, даже если в таблице Б.....
1. Показать всех клиентов, даже если у них нет заказов
2. Показать все товары, даже те, которые ни разу не покупали

Найти то, чего нет во второй таблице. То если если мы соединим, то мы наглядно увидим у чего нету пары, какие группы ни разу не писали альбом, какие покупатели ничего не заказывали и тд, ведь напротив будет NULL.

Возьмём базу данных Библиотеки
![[Pasted image 20260601020420.png]]
![[Pasted image 20260601020425.png]]

```SQL
SELECT     
	m.name,
    l.book_title
FROM members as m

LEFT JOIN loans as l
	ON m.member_id = l.member_id
```
![[Pasted image 20260601020610.png]]

То есть мы взяли первую таблицу полностью (Левую), а вторую (Правую) взяли лишь book_title. И как тут видно, данные из левой таблице совсем не потерялись, то где нету пары в правой таблице просто заполнилось NULL. Если бы мы использовали INNER JOIN, то данные с Павлом и Игорем просто бы не попали в финальную таблицу.

RIGHT OUTER JOIN идентичная вещь, но только зеркальная, его почти не используют, тк LEFT банально куда читабельней и приятней в использовании.

#### FULL OUTER JOIN
![[Pasted image 20260601022842.png]]

Как уже можно понять по картинке, это полное сливание таблиц. Берётся вообще всё из каждой и в случае если чего то не сходится, то оно заполняется как NULL.

Когда он нужен?
- Когда нужен полный аудит. Нужно сравнение обоих таблиц и найти все расхождения: Что добавилось, что удалилось, что изменилось. Например, сравнить справочник клиентов в старой и новой системе.
- Объединение двух таблиц для полной информации, даже если будут дырки, мы их прекрасно заметим.

Пример. Есть две таблицы Сотрудники и закреплённое за ними оборудование:
![[Pasted image 20260601024045.png]]
![[Pasted image 20260601024051.png]]

```SQL
SELECT em.name,
	   ep.device
	   
FROM employees as em

FULL OUTER JOIN equipment as ep
	ON em.emp_id = ep.emp_id
```

Как видно, у Ольги 2 оборудования, у Дмитрия и Сергея их вообще нет, а принтер никем не занят
![[Pasted image 20260601024248.png]]

В вывод попали вообще все строчки с обоих таблиц. А тут уже прекрасно видно, что Дмитрий и Серёга без оборудования, а принтер лежит без дела.

Так же очень важно подметить, что FULL OUTER JOIN не везде есть, например его нету в MySQL и он пишется как бы через объединение LEFT и RIGHT джоины, выглядит это как то так:

```SQL
SELECT e.name, eq.device
FROM employees e
LEFT JOIN equipment eq ON e.emp_id = eq.emp_id

UNION

SELECT e.name, eq.device
FROM employees e
RIGHT JOIN equipment eq ON e.emp_id = eq.emp_id;
```

#### SELF JOIN 

Это обычное объединение таблицы с самой же. Зачем это нужно? В любой задаче, где связь между строками находится внутри одной таблицы. Самый частный пример - иерархия сотрудников и начальников.

Так и будем показывать пример. Сотрудники и Начальники.
Представьте таблицу `employee`. В ней есть колонка `manager_id`, которая ссылается на `employee_id` начальника **в этой же таблице**.
![[Pasted image 20260601030259.png]]
```SQL
SELECT
    e.name AS Сотрудник,
    m.name AS Начальник
FROM
    employee AS e      -- Таблица "Сотрудники"
INNER JOIN
    employee AS m      -- Та же таблица, но теперь она "Начальники"
    ON e.manager_id = m.employee_id;
```

Как это работает?
1. **`employee AS e`** — мы говорим: «Это таблица сотрудников».
2. **`employee AS m`** — мы говорим: «А это таблица начальников». Физически это одна таблица, но логически мы разделили ее на две роли.
3. **`ON e.manager_id = m.employee_id`** — условие: «Возьми ID начальника из строки сотрудника и найди строку в таблице начальников, где такой ID есть».
SQL проходит по каждой строке в роли «сотрудника», берет его `manager_id`, ищет совпадение в роли «начальника» и подставляет имя.
![[Pasted image 20260601030459.png]]

## Операторы множеств - UNION, EXCEPT (MINUS), INTERSECT
#### UNION
UNION ALL - это объединение множеств. Это аналог concat из пандаса, когда у нас есть просто идентичный таблицы по колонкам, но это почему то две разные таблицы, то мы объединяем их при помощи UNION. 

Но тут есть небольшая разница в написании:
1. UNION ALL - склеиваем две таблицы и ему плевать на всё, он оставляет даже дубликаты в случае чего
2. UNION - если писать просто так, то он отбросит все дубликаты.

Пример работы данной операции:
![[Pasted image 20260601194608.png]]

```SQL
SELECT *
FROM A

UNION ALL

SELECT *
FROM B
```

Тут важно заметить, склеика идёт именно двух запросов, а не просто таблиц. То есть если у нас одна из таблиц даже немного отличается от второй, то мы всегда можем через запрос сделать её подобной, а потом уже склеить.
Название же колонок берётся только ИЗ ПЕРВОГО SELECT.

Так же при помощи данного оператора можно допустим вывести сразу количество строк многих таблиц.
```SQL
SELECT "album" AS table_name, count(*)
FROM album

UNION ALL

SELECT "band" AS table_name, count(*)
FROM band

UNION ALL

SELECT "person" AS table_name, count(*)
FROM person
```

## ОПЕРАТОР IF В SQL (CASE)

Данный оператор подобный тому, что мы вызываем в других языках программированиях, а именно вот эти IF ELIF ELSE. Сам CASE строится блоком, и если в случае IF в других языках, после условия следует какое то действие, то тут следует выражение, значение. То есть это допустим можно использовать для расставления флажков:
```SQL
SELECT album_id, name, band_id
	CASE WHEN band_id = 93 THEN 1 ELSE 0 END AS b_93
	CASE WHEN band_id = 192 THEN 1 ELSE 0 END AS b_192
FROM album as a
```

Как тут видно у данного выражения обязательно должен быть END. А вся конструкция выглядит как:
1. СASE - заводим весь процесс, далее 
2. WHEN - это условие которые мы ставим, 
3. THEN - Какое значение на выходе если условие сошлось
4. ELSE - Значение, если условие не сошлось.
5. добавляем Алиас, который будет выводится как отдельная колонка
![[Pasted image 20260601224937.png]]

Так же его можно использовать не просто для флажков, к примеру. Мы хотим посчитать количество определённый айди в альбоме, сколько у нас банд с айдишником 93 и сколько банд с айдишником 192, а так же общую сумму всех айди. Мы хотим вывести это всё рядом. Конечно можно это сделать при помощи Группировки и вывода количества, а потом допустим сделать UNION чтобы добавить сумму всего, но можно проще, вот пример:
```SQL
SELECT SUM(CASE WHEN band_id = 93 THEN 1 ELSE 0 END) as b_93,
	   SUM(CASE WHEN band_id = 192 THEN 1 ELSE 0 END) as b_192,
	   COINT(*) AS TOTAL
FROM album
WHERE band_id IN (93, 192)
```
Вот пример, интересный запрос, на всё воля фантазии.

## ОКОННЫЕ ФУНКЦИИ (говорят они важнее всего, пора выяснить)

Оконные функции или Аналитические функции, используются редко конечно, но они очень полезны и часто могут заменить огромные запросы на лишь небольшой вызов. Во многих вещах они очень удобны

Внешне оконные функции очень похожи на обычные функции агрегации
`SUM(n_albums) OVER (...)` - Тут сначала идёт обычная агрегация данных, а уже далее идёт `OVER`
в котором указываются дополнительные настройки для суммы. Рассмотрим один из сценариев.

Как мы уже прекрасно знаем, мы не можем вывести в одной таблице Детальные данные нашей таблицы и агрегированные данные. Потому что когда производишь агрегацию, ты обязан в SELECT писать именно то, что ушло у тебя в GROUPBY + какие то функции агрегации которые в дальнейшем можно отсортировать при помощи HAVING. Но добавлять туда другие колонки, которые ты не группировал или агрегировал нельзя. Тут то на помощь и проходят оконные функции, они позволяют решить этот вопрос и создать запрос Детальных данных + агрегация.

У нас есть группа `band_extended`, в самой таблице есть количество альбомов написанных каждой группой
![[Pasted image 20260602145037.png]]
Но мы хотим посчитать проценты альбомов каждой группы от всех альбомов в наших данных. Во первых нам надо посчитать общее кол-во всех альбомов. Тут важно, это число надо вывести именно в отдельную колонку, а не просто одну
![[Pasted image 20260602145119.png]]
И Далее мы уже поделим кол-во альбомов на тотал.
Допустим чтобы найти тотал можно написать базовый запрос:
```SQL
SELECT SUM(n_albums) as total
FROM band_extended
WHERE n_albums IN (303, 93, 192)
```

Но данный запрос выводит лишь 1 строку, и мы не можем добавить сюда обычные данные, ведь у нас в select уже есть агрегация. Ладно допустим напишем запрос отдельно, которые будем выводить все наши данные:
```SQL
SELECT band_id, name, year, n_albums
FROM band_extended
WHERE n_albums IN (303, 93, 192)
```

Объединим теперь наши запросы, используя первый как подзапрос в CROSS JOIN:
```SQL
SELECT d.band_id, d.name, d.year, d.n_albums, a.total,
		d.n_albums * 100.00 / a.total as pct_albums
FROM band_extended as d
CROSS JOIN (
			SELECT SUM(band_id) as total
			FROM band_extended
			WHERE band_id IN (303, 93, 192)
			) as a
WHERE d.band_id IN (303, 93, 192)
```

Выглядит не особо красиво честно говоря. Всякий раз когда мы пишем один запрос 2 раза, первый раз для самих данные, а второй для их суммы, то скорее всего такой запрос можно переписать оконными функциями. Теперь давай же посмотрим как это будет выглядеть с оконными функциями:

```SQL
SELECT band_id, name, year, n_albums,
		SUM(n_albums) OVER() as total,
		n_albums * 100.00 / SUM(n_albums) OVER () as pct_albums
FROM band_extended
WHERE band_id IN (303, 93, 192)
```

Во второй паре круглых скобок мы ничего не указываем, но так там можно показать какие именно СТРОКИ мы хотим суммировать.

Конечно туда можно добавлять не только SUM, но и все другие различные агрегированные данные:
![[Pasted image 20260602150839.png]]

Теперь поговорим о неком аналоге группировки, но только для оконных функций, а именно PARTITION BY.  Допустим у нас есть вот такие таблицы
![[Pasted image 20260602151952.png]]

И у нас стоит задача. Для каждого альбома, найти кол-во лет которое прошло с момента выхода первого альбома, до выхода текущего альбома.
![[Pasted image 20260602152202.png]]
```SQL
SELECT album_id, name, band_id, year,
		MIN(year) OVER(PARTITION BY band_id) as min_year,
		year - MIN(year) OVER(PARTITION BY band_id) as years_since_1st
FROM album
WHERE band_id IN (93, 192)
```

Тут мы уже указываем в OVER(PARTITION BY band_id) - мы как бы говорим, что произведи небольшую группировку колонки band_id и найди сумму каждой группы отдельно.

#### Кумулятивные функции (Накопительные суммы, накопительный итог)

Допустим у нас есть вот такая крутая таблица - ![[Pasted image 20260602153826.png]]
И мы желаем посчитать кумулятивную сумму по группам и годам, она показана на последней колонке. Это накопительная сумма. То есть первый год, плюс второй, потом плюсуется третий и тд. Так по каждой банке. Чтобы это сделать нужно вот такое выражение:
```SQL
SELECT album_id, name, band_id, year, total_sales,
		SUM(total_sales) OVER(PARTITION BY band_id
						ORDER BY year
						ROWS BETWEEN UNBOUNDED PRECEDING
							 AND CURRENT ROW) AS cumilative_sales
FROM album_sales
WHERE band_id IN (93, 192)
ORDER BY band_id, year;
```

Как мы видим мы сортируем данные прямо в OVER() и это оочень важно для накопительных сумм, ведь нам очень важно, какой из них когда именно вышел. 
ROWS BETWEEN - это мы как раз создаём то самое ОКНО, какие строки участвуют в агрегации данных, создаём окно для функции, откуда она может брать информацию. Здесь буквально написано - СТРОКИ МЕЖДУ (НИЖНЯЯ ГРАНИЦА ДИАПАЗОНА И ВЕРХНЯЯ)
Нижняя граница - Все предыдущие строки
Верхняя граница - Текущая строка

Поговорим об особенностях ORDER BY в оконных функциях: ORDER BY в оконных функциях может отличаться от глобального. Для проверки финальный ORDER BY лучше делать таким же как и в функции

Надо быть очень аккуратным с NULL значениями при оконных функциях, ну с этими друзьями всегда надо быть аккуратным.
![[Pasted image 20260602155723.png]]
Как видно на рисунке, в зависимости от того где будет находится null, то от этого и будет зависеть вся сумма на каждом альбоме. Правильного варианта тут нет, всё зависит от природы данных.
Для этого сортируем null отдельно от всего в конце запроса, а именно
`ORDER BY a NULLS LAST` или `ORDER BY a NULLS FIRST`

#### Скользящие средние, предыдущие значения

До этого мы писали
```SQL
ORDER BY year
ROWS BETWEEN UNBOUNDED PRECEDING
		   AND CURRENT ROW
```

Это означает берём Всё что раньше + текущую строку. Но данный синтаксис можно поменять. Допустим взять Три предыдущие строки + текущая строка

```SQL
ORDER BY year
ROWS BETWEEN 3 PRECEDING
			AND CURRENT ROW
```

Либо взять 3 предыдущие и 3 последующие
```SQL
ORDER BY
ROWS BETWEEN 3 PRECEDING
			AND 3 FOLLOWING
```

Тут можно придумать очень много всякого интересного, это и есть скользящие средние значения.

#### Нумерация и ранжирование строк

ROW_NUMBER() - производит базовую нумерацию строк сверху вниз. Базовый пример:
![[Pasted image 20260602165017.png]]

У нас есть такая таблица и мы хотим выдать каждому альбому значение, как бы уровень по его продажам, кто первый, кто второй и тд. Тут то нам и поможет наша функция ROW_NUMBER:
```SQL
SELECT album_id, name, band_id, year, total_sales,
		ROW_NUMBER() OVER(ORDER BY total_sales DESC, album_id) as rn
FROM album
WHERE band_id = 2454
ORDER BY total_sales DESC, album_id
```

А если мы хотим сделать ранжирование, допустим дать медали по продажам. Тут есть неувязочка для ROW_NUMBER, а именно то, что есть 2 альбома, которые получили одно и тоже число продаж, кому из них что дать. Для этого на помощь и приходится ранжирование, а именно RANK():
```SQL
SELECT album_id, name, band_id, year, total_sales,
		ROW_NUMBER() OVER(ORDER BY total_sales DESC, album_id) as rn,
		RANK()       OVER(ORDER BY total_sales DESC) as rank
FROM album
WHERE band_id = 2454
ORDER BY total_sales DESC, album_id
```

И вот уже этот красавчик нам выдаст:
![[Pasted image 20260602165829.png]]

Но тут беда, что у нас после 3 сразу идёт 5, это может быть не особо хорошо, но для этого есть другая функция, а именно DENSE_RANK. Всё работает точно так же, но если были одинаковые ранги, то далее он не пропускает число.

# Полёты из Парижа

В какие города можно улететь из Парижа (Paris) и сколько времени это займёт?

Используйте конструкцию "as flight_time" для вывода необходимого времени. Это необходимо для корректной проверки.

Формат для вывода времени: HH:MM:SS

Поля в результирующей таблице:

```SQL
SELECT town_to,
        TIMEDIFF(time_in, time_out) as flight_time

FROM Trip
WHERE town_from = 'Paris'
```

# Компании с рейсами из Владивостока

Какие компании организуют перелеты из Владивостока (Vladivostok)?

Поля в результирующей таблице:

```SQL
SELECT C.name
FROM Trip as T
INNER JOIN Company as C
        ON T.company = C.id
        
WHERE T.town_from = 'Vladivostok'
```
